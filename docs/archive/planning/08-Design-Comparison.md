## 1. 목적
- `docs/design`와 `docs/planning`의 설계를 비교 평가하여, 현재 시점에서 더 우수한 설계 접근을 판단한다.
- 설계 품질, 실행 가능성, 유지보수성 관점에서 우열을 도출한다.

---

## 2. 비교 대상 요약

### 2.1 docs/design
- 아키텍처, 워크플로우, 에러 처리, 로깅, 설정, 에이전트 역할 등 **실행 레벨 설계** 중심.
- 문서가 주제별로 분리되어 있고, index를 통해 **읽기 경로가 명확**함.
- 서브 에이전트 프롬프트/행동 규칙 등 **구현 디테일**이 포함됨.

### 2.2 docs/planning
- PRD/TRD, 사용자 흐름, DB 설계, 디자인 시스템, 코딩 컨벤션 등 **기획 + 구현 초안** 중심.
- 이해관계자(기획/개발)와의 합의 및 범위 정의에 유리.
- 제품 목표/성과 지표/범위가 명확히 포함됨.

---

## 3. 평가 기준
| 기준 | 설명 |
|------|------|
| 목적 적합성 | 설계 문서가 실제 구현을 바로 지원하는가 |
| 완성도 | 주요 시스템 요소가 빠짐없이 기술되어 있는가 |
| 실행 가능성 | 의사결정 없이 바로 구현 가능한 수준인가 |
| 일관성/연결성 | 문서 간 참조와 흐름이 자연스러운가 |
| 리스크 대응 | 실패/예외/품질 관리까지 설계가 포함되는가 |
| 유지보수성 | 구조화/분리/확장성이 좋은가 |

---

## 4. 비교 결과

| 평가 항목 | docs/design | docs/planning |
|-----------|-------------|---------------|
| 목적 적합성 | **높음**: 구현 중심, 단계별 실행 흐름 명확 | 중간: 기획/요건 중심이라 구현 디테일 부족 |
| 완성도 | **높음**: 에러 처리, 로깅, 에이전트 정의, 체크리스트 포함 | 중간: 핵심 요구사항 중심이나 운영/오류 흐름 부족 |
| 실행 가능성 | **높음**: 바로 개발 가능한 수준 | 중간: 상세 정책/로직 부족 |
| 일관성/연결성 | **높음**: index 기반 문서 구조, cross-reference 명확 | 중간: 문서 간 연결은 있으나 설계 흐름이 분산됨 |
| 리스크 대응 | **높음**: 실패 시 전략, 재시도, 부분 성공 등 상세 | 낮음: 리스크 언급은 있으나 실행 규칙 부재 |
| 유지보수성 | **높음**: 모듈화된 문서 구성 | 중간: 범용 기획 문서 구조로 확장성 한계 |

---

## 5. 최종 평가 (우수 설계)

**결론: `docs/design`가 전반적으로 더 우수한 설계 문서다.**

**판단 근거:**
- 구현 단계에서 필요한 **행동 규칙, 단계별 제어, 에러 처리, 품질 검증**이 구체적으로 명시되어 있음.
- 문서 구조가 모듈화되어 있고 index 중심으로 **탐색/유지보수에 유리**함.
- 에이전트 중심 시스템에서 필수적인 **역할 정의와 실행 순서**가 명확함.

`docs/planning`은 제품 요구와 UX/범위를 잘 정리했으나, 개발 실행을 위해서는 추가 설계가 필요한 상태로 평가된다.

---

## 6. 보완 제안 (docs/design 중심)

### 6.1 planning → design로 흡수해야 할 강점 (보완 대상)
- PRD의 목표/성과 지표/범위 정의
- 사용자 페르소나/스토리 기반의 우선순위
- 디자인 시스템 규칙 및 UI 톤앤매너

---

## 7. 권장 액션 (docs/design 중심)
1. `docs/planning`의 핵심 요구사항을 `docs/design` 상위 문서(architecture/workflow)에서 참조하도록 연결.
2. `docs/design`에 목표/성과 지표/범위, 페르소나/우선순위, 디자인 시스템을 요약 추가.
3. 최종 설계 기준 문서를 `docs/design`로 단일화하고, `docs/planning`은 제품/전략 문서로 역할 분리.

## 1. 문제 정의

### 현재 상황
- 기획자가 목업(MVP)을 직접 만들지만, **목업만으로는 개발자에게 개발 요청을 할 수 없음**
- 개발팀과 커뮤니케이션하기 위해 **정책정의서, 화면정의서, 프로세스 흐름**을 수작업으로 작성
- 이 과정에 **2~3일** 소요되어 기획자의 핵심 업무인 전략적 아이디어 도출에 집중하지 못함
- 바이브 코딩에 익숙하지 않은 기획자는 개발팀과의 소통에 추가적인 어려움 겪음

### 사용자의 고통 포인트 (Pain Points)
1. **시간 낭비**: 반복적인 기획서 작성에 2~3일 소요
2. **일관성 부족**: 수작업으로 인한 문서 간 불일치
3. **소통 비용**: 개발팀이 목업만으로는 개발 불가, 추가 질문 발생
4. **SPA 크롤링 불가**: 기존 도구들은 SPA(React/Vue/Next.js) 목업을 크롤링하지 못해 직접 캡처해야 함

---

## 2. 목표

### 핵심 목표
> **목업 → 기획서 변환 시간을 2~3일에서 30분으로 단축하여, 기획자가 반복 작업에서 해방되고 전략적 아이디어에 집중할 수 있게 한다.**

### 구체적 목표 (SMART)
1. **Speed**: 기획서 작성 시간 **2~3일 → 30분** 단축 (96% 시간 절감)
2. **Consistency**: 화면 정의서와 정책 정의서의 **일관성** 보장
3. **Usability**: 바이브 코딩에 익숙하지 않은 기획자도 **쉽게 사용 가능**
4. **SPA Support**: React/Vue/Next.js 기반 SPA 목업도 **자동 크롤링 가능**

## 3. 사용자 페르소나

### 주요 페르소나: "기획자 김"
| 속성 | 내용 |
|------|------|
| **직업** | 내외부 서비스 기획자 |
| **경력** | 3~5년차 |
| **기술 수준** | 바이브 코딩 미숙, 개발팀 커뮤니케이션 어려움 |
| **사용 환경** | 사무실 PC, 업무 시간 중 |
| **목표** | 빠르게 일관된 기획서를 만들어 개발팀에 전달 |
| **니즈** | 기술적 지식 없이도 URL만 입력하면 자동으로 기획서 생성 |
| **고통** | 목업 검증 후 개발팀 넘기기 전, 2~3일 동안 수동으로 기획서 작성 |

## 4. 사용자 스토리

### Epic 1: 크롤링 및 데이터 수집 (FEAT-1)
> **목업 URL만 입력하면 모든 페이지를 자동으로 수집하고 싶어**

| 우선순위 | 사용자 스토리 | 인수 조건 |
|----------|--------------|-----------|
| P0 | URL 입력 후 SPA 포함 모든 페이지 크롤링 | - React/Vue/Next.js 라우팅 따라감<br>- 수동 URL 입력 가능<br>- 최대 50페이지/최대 깊이 5 |
| P0 | 각 페이지 자동 스크린샷 캡처 | - 화면 잘림 없이 전체 캡처<br>- 파일명: `화면ID_순서.png` |

### Epic 2: AI 기획서 생성 (FEAT-2)
> **수집된 정보로 정책/화면/프로세스 정의서를 자동으로 만들어줘**

| 우선순위 | 사용자 스토리 | 인수 조건 |
|----------|--------------|-----------|
| P0 | 정책정의서 자동 생성 | - POL-001, POL-002... ID 부여<br>- 정책 이름, 설명, 대상, 내용 포함 |
| P0 | 화면정의서 자동 생성 | - SCR-001, SCR-002... ID 부여<br>- 스크린샷, UI 요소, 동작 설명 포함 |
| P0 | 프로세스 흐름 자동 생성 | - 화면 간 이동 경로 시각화<br>- 정책 ID 참조 |
| P1 | 용어 사전 (Glossary) 생성 | - 프로젝트 내 주요 용어 정의 |

### Epic 3: PPT 내보내기 (FEAT-3)
> **생성된 기획서를 PPT 형식으로 다운로드하고 싶어**

| 우선순위 | 사용자 스토리 | 인수 조건 |
|----------|--------------|-----------|
| P0 | PPT 자동 변환 및 다운로드 | - 정책/화면/프로세스/용어 포함<br>- 스크린샷 이미지 포함<br>- `.pptx` 형식 |

## 5. 성공 지표

### 노스스타 지표 (North Star)
> **"기획서 작성 시간 2~3일 → 30분 실제 단축"**

### 입력 지표 (Leading Indicators)
| 지표 | 현재 | 목표 | 측정 방법 |
|------|------|------|-----------|
| 크롤링 성공률 | 0% | 90%+ | SPA 목업 50페이지 중 45페이지 이상 수집 |
| 스크린샷 품질 | 수동 캡처 | 화면 잘림 0% | 전체 페이지 중 잘린 페이지 0% |
| 기획서 생성 소요 시간 | 2~3일 | 30분 내 | URL 입력 → PPT 다운로드까지 |
| 사용자 만족도 | - | "다시 쓰겠다" | 프로토타입 사용 후 질문 |

## 6. 범위 (Scope)

### 포함 (In Scope)
| 항목 | 내용 |
|------|------|
| **핵심 기능** | URL 자동 크롤링, 스크린샷 캡처, AI 기획서 생성, PPT 내보내기 |
| **지원 대상** | SPA(React/Vue/Next.js) 포함 모든 웹 목업 |
| **실행 환경** | 각 기획자 로컬 PC에서 독립 실행 |
| **데이터 저장** | 파일시스템 (JSON/CSV/Markdown) |
| **UI 스타일** | Duolingo처럼 밝은 색감, 친절한 가이드 |
| **말투** | 전문적이지만 부드럽게 |

### 범위 외 (Out of Scope) — v1.0
| 항목 | 이유 | 향후 계획 |
|------|------|-----------|
| 내부 공용 서비스 | v1은 로컬 실행에 집중 | v2+ 고려 |
| 다양한 PPT 템플릿 | MVP는 단일 템플릿 | v2 추가 |
| 버전 관리 | 1인 운영, 복잡도 ↓ | v3+ |
| 협업 기능 | 독립 실행에 집중 | v3+ |
| 클라우드 배포 | 내부 도구, 보안 ↑ | 필요 시 고려 |

## 7. 가정과 리스크

### 가정 (Assumptions)
| ID | 가정 | 영향 |
|----|------|------|
| A1 | 목업 URL이 접근 가능하고 공개되어 있음 | 크롤링 가능 |
| A2 | PRD/SDD/README 파일이 텍스트로 분석 가능 | LLM 분석 가능 |
| A3 | 기획자가 Chrome/Edge 브라우저 사용 | Puppeteer 호환 |
| A4 | 로컬 PC에 최소 4GB RAM 여유 있음 | 크롤링/LLM 동시 실행 |

### 리스크 (Risks)
| ID | 리스크 | 확률 | 영향 | 완화 방안 |
|----|--------|------|------|-----------|
| R1 | SPA 크롤링 실패 (라우팅 복잡) | 중 | 높 | Tier 1: DOM `<a>`/`<Link>`/`sitemap.xml`<br>Tier 2: `onClick` 핸들러<br>Tier 3: 수동 URL 입력 |
| R2 | 스크린샷 화면 잘림 | 중 | 중 | Puppeteer full-page screenshot<br>수동 재캡처 기능 |
| R3 | LLM 생성 품질 저조 | 중 | 중 | Few-shot 프롬프트<br>사용자 피드백 반영 |
| R4 | 30분 내 처리 불가 | 저 | 중 | 병렬 처리 (Phase 3-1)<br>조기 종료 (최소 10페이지 확보 시) |
| R5 | 벤더 락인 (GLM/Gemini) | 저 | 저 | 추상화 레이어<br>다중 LLM 지원 |

## 8. 해결 가설 (Hypothesis)

### 가설
> **"목업 URL + 기획 문서를 입력하면, SPA 크롤링 + 스크린샷 + LLM 분석을 통해 30분 내에 일관된 기획서 PPT를 생성할 수 있다."**

### 검증 방법
1. **실험**: Todo MVP 10페이지로 프로토타입 개발
2. **관측**: 크롤링 성공률, 스크린샷 품질, 생성 시간 측정
3. **학습**: 실패 지점 분석 → Tier 기반 크롤링/수동 입력 추가
4. **다음 가설**: 기획자 피드백 후 UI 개선
